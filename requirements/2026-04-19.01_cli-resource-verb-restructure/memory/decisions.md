2026-04-19 — architect — ReviseCommand signature revised to revise(String taskId, String planId): Integer (was revise(String taskId)). Verified DefaultRefinerEngine.java:68 generates per-plan sequential ids (`task-%03d`) — task ids are NOT globally unique, so the bare-task-id signature would silently revise the wrong plan's task. New semantics: planId null/blank → use latest plan (matches R012's get-tasks default and preserves the kubectl-style `revise task <id>` UX); planId non-blank → must resolve to an existing plan. Failure modes: no plan exists, supplied plan unknown, task id absent from resolved plan.
  why: ASSUMPTION #4 in REQUIREMENT.md (task ids are globally unique) is wrong as-stated; carrying planId in the contract is the safer default until the analyst revisits. ReviseCmd impl already injects refinementPlanStore so no dependency-surface change was needed; only the signature moved.
