2026-04-29 — analyst — Stack: LangChain4j (in-process Java). No LangGraph in this iteration.
  why: keep first real generator simple; multi-step graphs / validation loops belong to future iterations.

2026-04-29 — analyst — Default provider is Gemma via LM Studio (OpenAI-compatible HTTP surface). Provider must be swappable via LangChain4j configuration without code changes.
  why: enables local-first development; cloud providers (Anthropic, OpenAI) become a configuration switch later.

2026-04-29 — analyst — Strategy is renamed from `lemma-absence-mvp` to `lemma-absence-llm` so `StrategyId.name` on every persisted proposal makes provenance obvious.

2026-04-29 — analyst — Failure categories surfaced at the behavioral level: LLM_UNREACHABLE, LLM_TIMEOUT, LLM_AUTH_FAILED, LLM_RESPONSE_EMPTY, LLM_RESPONSE_MALFORMED, LLM_OTHER. These map onto the existing ProposalStrategyFailedException reason field (FEAT-LAPS R015).
  why: user explicitly requested finer granularity than the current single message; the port contract already carries a reason string so no architectural change required.

2026-04-29 — analyst — Prompt wording is intentionally NOT pinned in the requirement. Only its responsibilities and the inputs it must surface are specified.
  why: prompt iteration is implementation work; pinning text would force a requirement change for every prompt tweak.

2026-04-29 — analyst — Output shape demanded from the LLM is a strict JSON object {quizSentence, translation}. Adapter does NOT validate the FEAT-QSENT grammar — that is the deriver's job (FEAT-LAPS R012). Adapter only validates JSON well-formedness and presence/non-emptiness of both fields.
  why: keeps adapter scope narrow; downstream parser owns DSL validation.

2026-04-30 — analyst — REQUIREMENT.md extendido con 3 reglas nuevas pedidas por @qa-tester (escalations/2026-04-30_qa-to-analyst.md): F-LAGEN-R012 (modo canned: idempotente, independiente del contexto, sin proveedor), F-LAGEN-R013 (seleccion de modo: default dinamico, canned opt-in), F-LAGEN-R014 (validacion atomica de la configuracion al iniciar el sistema). Severities: R012=minor, R013=critical, R014=major.
  why: las tres reglas son la fuente funcional para que @qa-tester pueda tagear handwrittenTests del CannedLemmaAbsenceQuizCandidateGenerator, DefaultLagenModeResolver y DefaultLagenConfigResolver sin violar la regla de no-transitive-coverage.

2026-04-30 — analyst — R014 vs extension de R008 — eleccion: regla nueva (R014). Razon: R008 declara *que perillas existen* y *que son tuneables sin recompilar*; R014 declara *cuando y como se valida lo tuneado*. Son promesas observables distintas (una sobre superficie de configuracion, otra sobre momento de validacion). Mezclar el detalle de atomicidad en el `<details>` de R008 ocultaria una regla con su propia severidad (major) y su propio test, ademas de violar el principio de reglas atomicas. R013 referencia R014 explicitamente para el caso "modo no reconocido", reforzando la separacion (R013 = espacio de modos validos, R014 = momento de la deteccion de invalidos).
  why: dar al QA un anchor unico y testable para "el resolver falla rapido y antes de tocar el curso", sin diluir R008.

2026-04-30 — architect — 3 contratos under-specified flaggeados por @qa-tester cerrados.
  Contrato 1 (classify mapping): tabla de 5 detectores en orden, evaluacion sobre la cadena getCause(). LLM_UNREACHABLE para ConnectException/UnknownHostException/NoRouteToHostException. LLM_TIMEOUT para SocketTimeoutException/TimeoutException. LLM_AUTH_FAILED para OpenAiHttpException con statusCode 401/403, fallback heuristico regex (?i)\b(unauthorized|forbidden|invalid api key|401|403)\b sobre getMessage() de la raiz. LLM_OTHER catch-all. NO se modela como Map estatico porque la logica no es tabla pura.
  Contrato 2 (providerIdFor): formato 'providerName:modelId' literal, sin normalizacion case/whitespace, rechaza con InvalidProviderIdException si provider o model contienen ':' (rompe round-trip de trazabilidad).
  Contrato 3 (defaults): materializados como record publico LagenDefaults(temperature=0.7, maxTokens=2048, timeout=Duration.ofSeconds(30)) en package lagen. QA puede asertar igualdad contra los getters en lugar de re-tipear literales.
  why: pinear los 3 contratos elimina ambiguedad para el QA refinement de nombres y para el developer; LagenDefaults como record evita literales magicos en tests.

2026-04-30 — architect — Bug de merge confirmado y reproducido en escala mayor: cuando emiti `_change: modify` sobre revision-infrastructure SOLO con sus packages internos, el merge BORRO los atributos top-level del modulo (description, dependsOn, allowedClients, uses), el modulo audit-cli ENTERO (incluyendo LagenMode, bootstrap.LagenModeResolver/LagenConfigResolver y sus excepciones), Y la enum LlmGenerationFailureCategory que vivia en revision-infrastructure.lagen. Se restauraron via patch correctivo re-emitiendo todos los elementos top-level. Tambien observe: `_change: modify` sobre dependencies aliases que ya existen en sentinel.yaml requiere `_change: modify` en el patch (no `add`) — mensaje de error es claro. Ademas, el merge de signatures dentro de una interface NO preserva el campo `throws:` cuando se reemplaza el array `exposes` — el `throws: [InvalidProviderIdException]` que intente agregar a providerIdFor no aparece en el patch final. No-blocker porque el handwrittenTest especifico cubre el contrato y `throws:` en sentinel es informativo.
  why: documentacion del patron de bug para futuros patches correctivos. Workaround robusto: SIEMPRE re-emitir el modulo entero con todos sus atributos top-level y todos los packages que esperas que sobrevivan, no confiar en merge a nivel modulo.
